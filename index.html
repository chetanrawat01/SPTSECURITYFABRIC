Section 01
1.Project Overview

SPT Security Fabric is represented here as the source technical document: a Salesforce access and permission security platform for administrators, security engineers, and compliance officers.

1.1 Introduction

SPT Security Fabric (SPTSF) is a Salesforce DX application built on the Lightning Platform that provides end-to-end visibility, analysis, comparison, and remediation of access control across a Salesforce organization. It surfaces who can access what, quantifies over-privileged users and configurations, and lets administrators simulate, preview, and apply corrective changes from a single unified console.

The application is designed as an internal security tooling suite for Salesforce administrators, security engineers, and compliance officers. It operates entirely within the Salesforce org — no external services, no external data movement — and leverages platform-native metadata (Profiles, Permission Sets, ObjectPermissions, FieldPermissions, SetupEntityAccess, SetupAuditTrail) as its primary data source.

1.2 Business Objectives

  • Provide a single pane of glass for Salesforce access management, replacing scattered Setup screens and manual spreadsheets.
  • Detect over-provisioning of dangerous system permissions such as Modify All Data, View All Data, and Manage Users.
  • Compute an objective, org-wide Security Score and Compliance Score to establish a defensible security baseline.
  • Enable side-by-side comparison of Profiles, Permission Sets, and Users to identify drift, redundancy, and mismatches.
  • Support inline remediation — administrators can grant, revoke, or synchronize permissions directly from the analysis UI.
  • Maintain an audit trail of security-relevant changes and store point-in-time security snapshots for trend analysis.

3.1 Architectural Overview

The application adopts a strict layered architecture designed around the constraints of the Salesforce platform, particularly governor limits, request-scoped execution, and the imperative-only nature of @AuraEnabled Apex methods.

Five layers make up the runtime: a Lightning Web Component (LWC) presentation layer, an @AuraEnabled controller layer, a domain service layer, a repository layer, and a data / cache layer. Cross-cutting concerns — wrappers (DTOs), caching, and async processing — sit outside the layers and are used by all of them.

3.1.1 Layer Responsibilities

Presentation (LWC)Renders UI, orchestrates user interaction, calls Apex imperatively.rolesPermissions, accessAnalysis, financialDashboard

Validates inputs, wraps exceptions into AuraHandledException, delegates to services.AccessAnalysisController, RolesPermissionsController, DashboardController, AuditController, UserAccessController, RiskAnalysisController, PermManagementController, PermissionComparisonController, AccessComparisonController, AccessManagementHubController
Service (Domain)Business rules, orchestration, cross-object logic. 3.1.2 Cross-Cutting Concerns

  • Data Transfer Objects (DTOs) — two wrapper classes centralize all @AuraEnabled DTOs: AccessAnalysisWrapper for the access analysis wizard and SPTSFWrapper for dashboard, risk, audit, and comparison payloads.
  • Caching — SPTSFCacheManager is a thin wrapper over the Platform Cache (Cache.Org.getPartition("local.SPTSFCache")) with sanitized keys, configurable TTLs, and prefix-based invalidation.
  • Async processing — a Schedulable (DailySecurityScanScheduler) triggers a stateful batch (SnapshotBatchJob) at 2 AM daily; a Queueable (PermissionAnalysisQueueable) is used for on-demand cache refreshes.
  • Security — every controller class is declared with sharing; DML uses Security.stripInaccessible with AccessType.UPDATABLE / AccessType.READABLE; validation and exception translation happens at the controller boundary.

5.2 Controller Layer

All controllers are declared with sharing and expose their public API via @AuraEnabled methods. Cacheable=true is used for pure read operations that benefit from Lightning Data Service (LDS) caching; cacheable=false is used everywhere a write occurs or a fresh read is required.

5.2.1 AccessAnalysisController

Primary controller for the multi-step access analysis wizard. Exposes source-selection methods (getProfiles, getPermissionSets), paginated component-list methods (getApexClasses, getVFPages, getCustomTabs, getCustomObjects, getFieldsByObject, getFlows, getLightningComponents, getGenericEntities), analysis methods (analyzeApexClassAccess, analyzeVFPageAccess, analyzeTabAccess, analyzeObjectAccess, analyzeFieldAccess, analyzeLightningComponentAccess, analyzeFlowAccess, analyzeGenericEntityAccess), and save methods (saveApexClassAccess, saveVFPageAccess, saveTabAccess, saveObjectAccess, saveFieldAccess, saveLightningComponentAccess, saveFlowAccess, saveGenericEntityAccess).

Design constraints enforced:
  • Profiles are always resolved to their owned PermissionSet before any DML or SOQL.
  • Mixing Profiles and Permission Sets in a single operation is rejected early.
  • Each component type (Class, VF Page, Tab, Object, Field, Flow, LWC) is completely independent; only the selected source IDs are shared.
  • Governor limits are enforced via pagination (safe page size and offset helpers) and via chunking of large ID sets.

5.2.2 RolesPermissionsController

5.3.3 Scoring & Weighting Rules

The security score is a penalty-based deduction from 100. Penalties are capped per-category to prevent any single signal from dominating the score.

SignalWeight per occurrenceCategory cap
PermissionsModifyAllData on profile-owned perm set3 pts15 pts
PermissionsViewAllData on profile-owned perm set2 pts10 pts
PermissionsManageUsers on profile-owned perm set2 pts8 pts
Inactive admin (no login > 90 days)5 pts15 pts

Standing bands: 90+ Excellent, 75-89 Good, 60-74 Fair, below 60 At Risk. The compliance score starts at 100 and deducts 4 points per non-profile permission set granting Modify All Data, with a floor of 60.

5.6 Asynchronous Processing

Three asynchronous mechanisms handle recurring maintenance and long-running work outside the synchronous request-response cycle.

5.6.1 DailySecurityScanScheduler (Schedulable)

Cron expression: 0 0 2 * * ? (2:00 AM daily). Named SPTSF Daily Security Scan. The scheduler aborts any existing job with the same name, invalidates the SecurityScore, RiskSummary, RiskyUsers, and Dashboard_Full cache entries, and enqueues a batch execution of SnapshotBatchJob with a scope size of 200.

5.6.2 SnapshotBatchJob (Batchable, Stateful)

Iterates active users in scope-200 batches. Each execute() slice tallies processedCount and errorCount on the stateful instance; finish() invalidates dependent cache entries so the next dashboard load reflects fresh data. Full snapshot serialization into Security_Snapshot__c is scaffolded and documented in the class for future implementation.

5.6.3 PermissionAnalysisQueueable

Fires on demand to invalidate risk-related cache entries following an inline remediation. Fast, side-effect-only work — no DML, no callouts.

5.7 Caching Strategy

The Platform Cache is used throughout the read path to avoid recomputing expensive aggregations on every dashboard load. Keys are sanitized (alphanumeric + underscore, max 50 chars) and stored in the local.SPTSFCache Org partition.

Cache KeyTTLConsumer
Dashboard_Full300 sDashboardAggregationService.getDashboard()
SecurityScore_Org600 sSecurityScoreService.calculate()
RiskSummary_Org300 sRiskEngineService.getSummary()
RiskyUsers_N300 sRiskEngineService.getTopRiskyUsers(N)

Invalidation is either explicit (SPTSFCacheManager.remove) or prefix-based (SPTSFCacheManager.invalidatePrefix), used by the scheduler and any write that could affect downstream aggregations.

Section 07
7.Security Design & Guarantees

Security guarantees cover sharing, CRUD/FLS enforcement, exception translation, governor safety, and cache behavior.

7.1 Sharing & Access Enforcement

  • Every Apex class is declared with sharing — enforcing record-level sharing rules for the running user.
  • CRUD & FLS are enforced via Security.stripInaccessible before DML and after SOQL where the returned data is serialized to the client.
  • All SOQL uses bind variables — no dynamic query concatenation with user input. String LIKE clauses are constructed with String.escapeSingleQuotes on the search term.
  • DML on User records checks Schema.SObjectType.User.isUpdateable() before executing the update.

7.2 Exception Handling Contract

The exception handling contract is uniform across all controllers. This makes error reporting predictable at the LWC boundary and prevents raw platform stack traces from leaking to the client.

LayerBehavior
Repository / ServiceThrows native exceptions (QueryException, DmlException, etc.) freely.
Controller (@AuraEnabled boundary)Wraps every non-AuraHandledException into AuraHandledException with a human-readable message. Rethrows AuraHandledExceptions unchanged so nested messages are preserved.
LWCReceives AuraHandledException.body.message and surfaces it via lightning/platformShowToastEvent.

7.3 Bulk Safety & Governor Limits

  • Pagination is enforced at every list endpoint — safePageSize and safeOffset helpers cap page size and offset to platform-safe values.
  • All SOQL is outside of loops; results are collected into maps for O(1) lookup.
  • The comparePermissions endpoint enforces a maximum of 5 Permission Sets per call.
  • Async work uses Queueable (single-record) or Batchable (bulk) depending on scale; the batch scope size is 200.

7.4 Cache Safety

  • Cache misses are non-fatal — every get() / put() call is wrapped in try/catch and degrades gracefully to a live SOQL execution.
  • Cache keys are sanitized to alphanumeric + underscore and truncated to 50 characters to comply with the Platform Cache API.
  • Cache invalidation is prefix-based, so a single write can invalidate all dependent read caches in one call.

code

9.3 Recommended Test Coverage Targets
